## 01. 프로젝트 개요 (Introduction)

### 한 줄 요약

**사용자가 질문을 입력하면 → 서버가 답을 찾아 → 채팅 화면에 보여 주는 사내 지식 챗봇(프로토타입).**

- 예: "건전지 어디 있어?" → Exact Match 응답 예: 탕비실 세 번째 서랍(건전지) **위치 안내**  
- 예: "탕비실에서 뭐 구할 수 있어?" → 저장 지식 기반 **AI 문장 생성 응답**

### 배경

- 신규 입사자의 반복적인 기초 정보(위치, 규정) 문의로 인한 기존 인력의 업무 몰입 저해  
- 사내 가이드 문서의 파편화로 정보 검색 시간 낭비  

### 목적

- "물어보기 미안한" 사소한 질문의 창구 단일화  
- 온보딩 프로세스의 자동화 및 효율화  

### 기대 효과

- 신규 입사자 적응 시간 단축  
- HR/총무팀의 반복 문의 응대 시간 절감

## 02. 서비스 벤치마킹 (무신사 '촤비스')

### 레퍼런스 분석

- **친숙한 메신저 UI:** 슬랙/카톡 스타일로 접근성 극대화  
- **하이브리드 답변:** 정확한 정보는 DB에서, 복합 질문은 AI가 처리  

### 본 프로젝트의 차별점

- **퍼블리셔 강점 활용:** React 활용한 더 유연하고 빠른 UI 인터랙션  
- **실시간 고도화:** 어드민에서 즉시 지식을 보강하는 '데이터 피드백 루프' 구현

## 03. 상세 기능 기획 (Main Features)

### 03-1. 챗봇 (User)

**하이브리드 응답 로직**

- **1단계(키워드):** "건전지", "주차" 등 단어 매칭 시 정확한 위치/절차 안내  
- **2단계(AI):** 키워드 부재 시 Ollama(Local LLM) 등이 사내 문서(검색 결과)를 기반으로 답변 생성  

**UI 포인트**

- 답변 하단 관련 사내 링크(사내 게이트) 자동 연동  
- 자주 묻는 질문 칩(Chip) 버튼 제공  

### 03-2. 어드민 (Admin)

**지식 베이스(Knowledge Base) 관리**

- 키워드–답변–링크 셋을 관리자가 직접 등록/수정 가능한 대시보드  

**미응답 질문 수집 및 고도화**

- 챗봇이 답변하지 못한 질문을 리스트업하여 '미답변 질문'으로 분류  
- 관리자가 해당 질문에 대한 답을 등록하면 즉시 챗봇 지능 업데이트  

### 03-3. 상세 기능 정리

- **실시간 미응답 수집 (Logging)**  
  - 내용: 챗봇이 답변하지 못한 질문을 자동 감지 및 저장  
  - 효과: 임직원이 궁금해하는 '정보의 공백'을 실시간 데이터로 파악  
- **자율적 지식 관리 (Admin)**  
  - 내용: 관리자 UI를 통한 지식 등록·수정·삭제 (No-Code 운영)  
  - 효과: 개발자 개입 없이 사내 변화(규정, 위치 등) 즉시 반영  
- **실시간 지능 업데이트 (Real-time)**  
  - 내용: 등록된 지식을 별도 서버 재시작 없이 즉시 서비스 적용  
  - 효과: 사용자가 물어볼수록 똑똑해지는 선순환 서비스 구조 구축

## 04. 기술 스택·시스템 구조·처리 플로우 (Technical)

### 04-1. 기술 스택

- **Frontend:** React.js (반응형 웹 및 애니메이션), Vite, Tailwind CSS  
- **Backend:** Node.js (Express), ES Modules — API·Ollama 중계 및 로직 처리  
- **Database:** ChromaDB — 벡터(의미) 검색용 경량 DB  
- **AI Engine:** Ollama (Local LLM) — 오픈 API 연동 및 테스트  

### 04-2. 시스템 구조도 (개념)

```
[User] ↔ [React UI] ↔ [Node.js Backend] ↔ [ChromaDB / Ollama]
```

### 04-3. 요청 흐름 (저장소 기준)

```
[사용자]
   │  브라우저에서 채팅 화면 열기 (예: 배포 URL 또는 로컬 localhost:5173)
   ▼
[React 앱 — client/]
   │  질문 입력 → 전송 → 서버에 POST 요청
   ▼
[Node 서버 — server/]
   │  ① 키워드 매칭? ② ChromaDB 검색? ③ Ollama에 답 생성 요청?
   │
   ├── [Exact Match용 JSON] ← 키워드 → 저장된 답·링크 즉시 반환
   ├── [ChromaDB] ← 의미가 비슷한 문장 검색
   └── [Ollama] ← 검색 문단 + 질문으로 답 문장 생성
   ▼
[React 앱] → 말풍선으로 표시 → [사용자]
```

- **client:** 사용자 채팅 UI  
- **server:** 질문 수신, 파일·벡터 DB·AI 조합으로 응답 생성  
- **ChromaDB:** 문장 저장 및 의미 유사 검색  
- **Ollama:** 로컬 LLM(Llama 등). 검색된 문단을 바탕으로 자연어 답변 생성  

### 04-4. 저장소·주요 파일

**프로젝트 루트 (`2026_01_chatbot/`)**

- **client/** — 채팅 화면(프론트엔드). React 기반.  
- **server/** — API 서버(백엔드). ChromaDB·Ollama 연동.  
- **README.md** — 설치·실행·API 요약.  
- **DEPLOY.md** — Vercel·Render 등 배포 가이드.  

**client/**

- **src/App.jsx** — 채팅 UI 중심(메시지, 입력, FAQ 칩, API 호출).  
- **src/index.css** — 전역 스타일·Tailwind.  
- **vite.config.js** — 개발 시 프론트(5173)→백엔드(3001) 프록시.  
- **package.json** — 프론트 의존성.  

**server/**

- **index.js** — Express 진입. `/api/chat`, `/api/knowledge`, `/api/unanswered` 등.  
- **ingest.js** — ChromaDB 초기·배치 적재 스크립트.  
- **data/exact-match-knowledge.json** — 키워드→답변·링크(Exact Match, DB 미사용).  
- **data/unanswered.json** — 미답변 로그.  
- **package.json** — 서버 의존성.  

### 04-5. ChromaDB (역할)

- **일반 DB:** 문자열 일치 위주 검색.  
- **ChromaDB(벡터):** 문장을 의미 벡터로 저장·조회. 다른 표현으로 질문해도 유사 문단 검색 가능.  

**역할 정리**

- **저장:** `ingest.js`가 지식 문단을 적재(임베딩: `@chroma-core/default-embed` 등).  
- **검색:** 서버가 질문을 넘겨 상위 N개 유사 문단 수신.  
- **실행:** 별도 프로세스 권장. 예: Docker `chromadb/chroma` 이미지, 포트 **8000**. Node는 해당 URL로 접속.  

**한 줄:** 질문과 의미적으로 가까운 사내 지식 문단을 찾는 데 사용.

### 04-6. Ollama (역할)

- **Ollama:** 로컬에서 구동하는 LLM 런타임. Llama 등 모델 탑재.  
- 서버는 ChromaDB에서 얻은 **문단 + 사용자 질문**을 넘기고, 한국어 답 문장을 생성하도록 요청.  

**역할 정리**

- ChromaDB: 참고 문단 선별.  
- Ollama: 문단을 반영한 **답 문장 생성**.  

**실행:** Ollama 앱 또는 `ollama run llama3` 등으로 로컬 서비스 기동 후 서버의 `OLLAMA_HOST` 등과 맞춤.  

**한 줄:** ChromaDB가 고른 문단을 재료로, 답변 문장을 만드는 AI 엔진.

### 04-7. 질문 1건 처리 순서 (구현 플로우)

1. 사용자가 채팅에서 질문 입력 후 전송.  
2. React가 `POST /api/chat`으로 `{ question: "..." }` 전송.  
3. **Node 서버**  
   - **① Exact Match:** `exact-match-knowledge.json`에서 키워드 포함 여부 확인 → 있으면 답·링크 반환 후 **종료**(Chroma/Ollama 미사용).  
   - **②** 없으면 ChromaDB에서 유사 문단 검색.  
   - **③** 검색 0건 → 사내 미등록 안내 등 반환. 필요 시 **미답변**으로 `unanswered.json`에 기록.  
   - **④** 검색 1건 이상 → 문단·질문을 **Ollama**에 전달해 RAG 형태 답변 생성.  
4. JSON 응답을 React가 말풍선으로 표시.  

**정리:** **Exact Match → ChromaDB 의미 검색 → (히트 시) Ollama 답변 생성.**

## 05. 로컬 실행·명령어·배포 참고 (Operations)

### 05-1. 기동 순서

1. **ChromaDB** (예: Docker) — 의미 검색 서버 필요.  
2. **지식 적재** `node ingest.js` (server/) — 컬렉션이 비어 있으면 검색 불가. 최초 1회·데이터 갱신 시.  
3. **Node 서버** `npm start` (server/) — API·파이프라인 본체.  
4. **React** `npm run dev` (client/) — 로컬 UI(예: 5173).  
5. **(선택) Ollama** — RAG·생성 응답에 필요. 미기동 시 해당 구간 오류 또는 폴백 메시지.  

**순서 요약**

```text
1) Docker: chromadb (8000)
2) server: node ingest.js  (필요 시)
3) server: npm start        (3001)
4) client: npm run dev      (5173)
5) (선택) Ollama
```

### 05-2. 자주 쓰는 명령어

- **ChromaDB 기동 예시:** `docker run -d -p 8000:8000 chromadb/chroma`  
- **지식 적재:** `cd server` → `node ingest.js`  
- **API 서버:** `cd server` → `npm start`  
- **채팅 UI(로컬):** `cd client` → `npm run dev`  
- **Ollama 확인:** `ollama run llama3`  

### 05-3. 배포 개요

- **프론트(client):** Vercel 등에 배포 → 사용자는 URL로 채팅 화면 접속.  
- **백엔드(server):** Railway, Render 등에 배포 → API 베이스 URL 설정.  
- 운영 환경에서는 Chroma를 **동일 호스트 Docker** 등으로 두거나, Ollama는 **별도 VM·외부 API**로 분리하는 구성이 일반적. 상세는 **DEPLOY.md** 참고.

## 06. 추진 일정 및 향후 계획 (Roadmap)

### 1단계: 기획 및 백엔드 인프라 구축 (1~2주차)

- **선작업:** Node.js 서버 환경 구성 및 Ollama 로컬 모델 연동 테스트  
- **ChromaDB 세팅:** 사내 지식 데이터를 벡터화하여 ChromaDB에 적재하는 파이프라인 구축  
- **협업:** 디자인팀 전소연 과장님과 UI/UX 컨셉 확정  

### 2단계: 프론트엔드 퍼블리싱 및 API 연동 (3~5주차)

- **주요 작업:** React + Tailwind CSS 기반의 채팅 및 어드민 UI 구현  
- **백엔드 연결:** Node.js 서버와 React 프론트엔드 간의 API 통신 구현  

### 3단계: AI 연동 및 RAG 로직 구현 (6~7주차)

- **주요 작업**  
  - **RAG 구현:** ChromaDB 검색 결과와 Ollama 모델을 결합한 답변 생성 로직 최적화  
  - **로직 최적화:** [키워드 매칭 → ChromaDB 검색 → AI 답변] 순 프로세스 고도화  

### 4단계: 안정화 및 최종 배포 (8주차)

- **주요 작업**  
  - 전체 시나리오 테스트 및 UI 디테일(마이크로 인터랙션) 폴리싱  
  - Vercel을 활용한 실시간 웹 배포 및 최종 점검  
  - 사내 시연을 위한 가이드 문서 및 발표 자료 완성  

### 향후 계획

- 사내 메신저(Teams 등) 연동 및 확장 가능성 검토
